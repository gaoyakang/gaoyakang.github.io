现在已经将data的数据绑定在vue的实例上了，接下来就需要把数据更新在页面上了，也就是要把文本节点的内容替换成我们修改后的data的数据，有两种方法：

  • 1.直接保存需要修改的节点(就是我们上面使用的方法)，然后将修改后的data数据直接赋值在保存的节点上，但当data中有很多的数据且页面上多处使用了数据，就需要去关心所有数据之间的对应的关系，保留各个节点的状态，实现数据的更新，

  • 2.只需在数据源发布数据，使用数据的地方订阅数据，打破这种数据之间的对应依赖关系的模式
    很明显需要使用第二种方式

它的流程就是：当input中的数据被修改时就会触发input事件，此时我们就需要把数据发布出去，但实质上这个修改的数据最终是要和data的text属性关联起来，所以直接把数据复制text，当text被赋值后就是在修改text的值，它的值被修改时就会触发text属性的set函数，在这里进行真正的数据发布，因为还需要在数据发布后订阅该数据，所以就把这两个功能封装成一个类Dispatch，在dispatch中它有add(订阅)和notify(发布)两个函数,这两个函数并不与数据直接接触，它们处理的实际上是一个又一个的观察者对象，在需要数据变化的地方就新建观察者对象，然后当订阅事件的时候，实际上订阅的就是观察者对象，在数据修改时，就可以调用观察者中的更新函数更新数据，这样发布与订阅的双方就能打破原有的联系，不需要数据改变时去先保留节点，只需要询问观察者即可，有变化就更改，无变化就保持

vue虚拟DOM和Diff算法

在上面实现了vue的双向数据绑定，但采用的是直接将劫持的数据渲染在真实的dom上，对于少量的dom操作来说还比较好用，但是当有大量的频繁的dom操作执行时，重复的去操作dom就会造成很大的浪费，比如说在vue中，使用v-for进行循环时，如果采用上面的这种每次数据变化就去改变dom，本来只有dom中的数据发生了改变，但却需要把数据所在的dom全改一遍，所以就需要一种只需要改变数据变化的部分的机制来处理这种情况，vue中采用的就是虚拟dom+diff算法来实现这一目标的，虚拟dom指的是，当数据在进行更改时，并不会直接将数据渲染在页面上，而是先把原本的dom结构抽象成一个对象，这个对象包含了dom中节点的所有内容，位置，属性，数据等等，然后当数据发生改变时，就会把改变的内容重新生成一个对象，然后整两个对象先进行比对，把不同的地方找出来，不同的地方就是最终要渲染在页面的dom，此时再进行渲染dom，就能减少消耗，这些比对的对象就是虚拟DOM，比对的方法就是Diff算法，比对的差异就是补丁Patch，那虚拟dom是怎么生成的呢？diff算法又是如何比对的呢？
